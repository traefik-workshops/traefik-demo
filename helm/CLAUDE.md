# Agent guide — `helm/`

Section-specific rules for Helm charts. Inherits everything from [`../CLAUDE.md`](../CLAUDE.md).

## Scope

Helm charts published to `oci://ghcr.io/traefik-workshops` and consumed by demo repos. Each chart is one app or composition.

## Chart layout (canonical)

```
helm/<name>/
├── Chart.yaml          # metadata (apiVersion v2, type application, version, appVersion, maintainers, sources, keywords)
├── Chart.lock          # generated by `helm dep update`, committed
├── values.yaml         # defaults — every leaf field has a comment
├── values.schema.json  # JSON Schema validating values.yaml (helm lint --strict enforces)
├── .helmignore         # standard exclusions + repo-specific (.DS_Store, .git, services/, files/private/)
├── ct.yaml             # chart-testing config (per TESTING.md)
├── README.md           # purpose, install snippet, values table (auto-generated between <!-- BEGIN_HELM_DOCS --> markers)
├── crds/               # optional — CRDs the chart owns (keycloak does this)
├── charts/             # populated by `helm dep update`; gitignored
├── files/              # optional — static files the templates `tpl` (airlines uses this)
└── templates/
    ├── NOTES.txt       # mandatory — post-install message
    ├── _helpers.tpl    # the canonical 5-helper set: .name, .fullname, .chart, .labels, .selectorLabels
    ├── deployment.yaml
    ├── service.yaml
    └── *.yaml          # other resources
```

All charts follow this convention as of the v4 sweep. The `new-chart` skill scaffolds it.

## Versioning model (read this carefully)

**Chart versions are slaved to the repo tag.** When the repo cuts `v3.3.0`, every `Chart.yaml`'s `version:` is rewritten to `3.3.0` in the same commit and every chart is published at that version. This is enforced by `make release-bug/feature/major`, which sweeps `helm/*/Chart.yaml` before tagging.

What this means in practice:

- **`Chart.yaml` `version`** = the repo tag, minus the `v` prefix.
- **`Chart.yaml` `appVersion`** = the upstream app's release. Moves independently. Don't add a `v` prefix unless the upstream uses one.
- **Subchart `version:` for in-repo `file://` deps** = also the repo tag. Sweeping handles this.
- **External subcharts** (e.g. weaviate) keep their upstream version.
- **Adding a chart in a patch release is fine** — it just gets the current repo version and gets published at that version.

This trades per-chart independence for cross-cutting simplicity. Consumers pin everything (Terraform modules AND Helm charts) to one repo tag. A patch in `airlines` ships with a patch in every other chart too — that's the deal.

## Chart.yaml conventions

- `apiVersion: v2` — always (Helm 3+).
- `name` — matches the directory: `helm/<name>/Chart.yaml` has `name: <name>`.
- `type: application` (use `library` only if the chart provides no resources, just templates).
- `version` — managed by the release flow; don't edit by hand.
- `appVersion` — quoted string matching the upstream app's release.
- `maintainers` — at least one (name + email). Required by `helm lint --strict`.
- `sources` — list of repo URLs (upstream source if any, plus this chart's URL).
- `home` — chart homepage (docs URL or this repo's URL).
- `keywords` — a few tags.
- `kubeVersion` — pin the minimum (`">= 1.27.0-0"` is a reasonable floor).
- `dependencies` — declare every subchart, with `condition` so callers can disable optional ones.

## values.yaml conventions

- **Comment every top-level group.** Agents and humans read values.yaml first; uncommented YAML is opaque.
- **Booleans use `enabled: true|false`** for feature toggles. Mirror the convention from the Terraform side.
- **Defaults are dev-grade.** Single replica, in-cluster databases, demo passwords. Production-grade knobs are exposed but off.
- **Never default a real secret.** Use an empty string + a `NOTES.txt` instruction to set it at install time (`--set <key>=<value>`). The chart's `helm lint --strict` will not pass without explicit values for required secrets.
- **Image tags are pinned**, not `latest`. `dns-traefiker` uses `latest` today — that's debt, not a pattern.
- **Resources are always set** — at least `requests`; `limits` if the chart isn't burstable.
- **securityContext + podSecurityContext** declared on every Deployment (drop ALL caps, readOnlyRootFilesystem when feasible, runAsNonRoot).

## values.schema.json conventions

- Generated by the `new-chart` skill from `values.yaml`; hand-edited to add descriptions.
- `additionalProperties: false` on the top level (catches typo'd value keys early).
- Required fields listed explicitly.
- `helm lint --strict` enforces it on every PR. If the schema rejects valid values, fix the schema, not the values.

## templates/ conventions

- **`_helpers.tpl` defines five things**: `.name`, `.fullname`, `.chart`, `.labels`, `.selectorLabels`. The `new-chart` skill generates this set; keep them consistent across charts so cross-chart wiring is predictable.
- **Every workload resource carries `{{ include "<chart>.labels" . }}`.** Selectors use `.selectorLabels`. Don't open-code labels.
- **`NOTES.txt` is mandatory.** Tell the demo author what they just got: URLs, admin credentials (with a "rotate this" reminder), and next-step CLI commands.
- **Templates that emit Traefik CRDs** (IngressRoute, Middleware, ServersTransport) gate on a value (`ingress.enabled`, `middleware.enabled`) so charts work in clusters without Traefik installed.
- **Helm tests** live under `templates/tests/` with `helm.sh/hook: test`. See `../TESTING.md`.

## Subchart dependencies

- **Declare in `Chart.yaml` → `dependencies`** with: `name`, `version` (matches repo tag), `repository`, `condition` (always).
- **Repository field** uses `file://../<name>` for sibling charts in this repo. `make package` rewrites these to `oci://ghcr.io/traefik-workshops` before pushing.
- **Sub-chart values** are nested under the dep's chart name (or `alias`).
- **Don't bypass conditions.** If a feature requires a subchart, gate the feature on the subchart's condition variable.

## When changing an existing chart

1. Skim `../helm/CLAUDE.md` for chart conventions before changing anything.
2. If renaming/removing a value, or changing a default → `release-major` (breaks consumers' values.yaml).
3. If adding a value, it must have a default — else minor becomes major.
4. Run `make check` before committing.

## When adding a new chart

Use the `new-chart` skill (`../.claude/skills/new-chart/`). If you're invoking it conversationally:

> "Scaffold a new chart called `<name>` for `<one-line purpose>`"

The skill writes the canonical shape, including pinning `Chart.yaml` `version:` to the current repo tag.

## Don't

- **Don't edit `Chart.yaml` `version:` by hand.** The release flow owns it. Editing it manually creates inconsistency between charts.
- **Do commit `Chart.lock`.** `make release-*` runs `helm dep update` and commits the fresh lock alongside the version sweep, so the packaged chart has deterministic subchart digests.
- **Don't commit `charts/*.tgz`.** They're regenerated from `Chart.lock` by `helm dep update`. `.gitignore` covers this.
- **Don't bypass `make check`.** If the lint catches something, fix the chart, not the lint rule.
- **Don't default a real credential.** Ever. Use a placeholder + NOTES.txt instruction.
- **Don't add a new chart-level dependency** (new subchart, additional CRD requirement) without flagging it in the PR.
- **Don't put application source code in `helm/`.** `helm/airlines/services/` is a historical artifact — don't propagate the pattern.
