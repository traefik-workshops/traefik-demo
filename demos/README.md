# demos/

Runnable, white-labeled **compositions** of the library in this repo. Four run
end-to-end on k3d for free (`make up` / `make scenarios` / `make down`); one
targets a cloud (`oidc-portal`, AWS). All of them double as the reference shapes
the `sa-assistant` / `build-poc` skill pattern-matches when generating a PoC. See
[`AGENTS.md`](./AGENTS.md) for the full convention.

Module sources are **relative** (`../../terraform/...`) so every demo tracks the
library in this checkout and `terraform validate` runs offline. (Standalone PoCs
generated by the `build-poc` skill pin `?ref=<version>` instead.)

## Demos

| Demo | What it shows | Modules pulled | CI |
|---|---|---|---|
| [`single-cluster`](./single-cluster) | One k3d cluster + Traefik Hub + whoami — the "hello world." | `compute/suse/k3d`, `traefik/k8s`, `apps/whoami/k8s` | deploy + scenarios |
| [`unified-ingress`](./unified-ingress) | Multicluster transit + app-workload — the dominant real-world shape. | `compute/suse/k3d` (×2), `traefik/k8s` (parent + child), `observability/opentelemetry/k8s`, `apps/whoami/k8s` | deploy + scenarios |
| [`ai-gateway-openai`](./ai-gateway-openai) | AI Gateway over an OpenAI-compatible backend: regex + Presidio content-guards and a token rate-limit. | `compute/suse/k3d`, `traefik/k8s`, `ai/presidio/k8s` | deploy + scenarios |
| [`hub-from-source`](./hub-from-source) | Run Traefik Hub built from your local source on k3d via the module's `custom_image_*` override — the dev loop for testing a Hub change. | `compute/suse/k3d`, `traefik/k8s`, `apps/whoami/k8s` | deploy + scenarios |
| [`oidc-portal`](./oidc-portal) | Traefik Hub API Portal + Cognito on EKS (swap for EntraID or Keycloak). | `compute/aws/eks`, `compute/aws/vpc`, `security/cognito`, `traefik/k8s`, `apps/whoami/k8s` | validate only (cloud) |

The four k3d demos are deployed and smoke-tested in CI
([`.github/workflows/demos-ci.yml`](../.github/workflows/demos-ci.yml), gated on a
`TRAEFIK_HUB_TOKEN` secret). `oidc-portal` is cloud (EKS + Cognito), so CI only
`terraform validate`s it — run it by hand. `ai-gateway-openai` is white-labeled
from `clients/Mercury`; `hub-from-source` from `k3d-unified-ingress`.

## Conventions

Every demo ships:

- `main.tf` — the module composition (the file an agent pattern-matches against)
- `variables.tf` — inputs the demo expects
- `terraform.tfvars.example` — placeholder values. **Never commit real values.**
- `outputs.tf` — dashboard / route URLs and any module pass-throughs
- `README.md` — what the demo proves, prereqs, run steps, how to extend

The k3d demos additionally ship `versions.tf` (incl. the `k3d` provider), a
`Makefile` (`up` / `down` / `scenarios` / `fmt` / `validate`), and `scenarios.sh`
(curl assertions vs an expected table). `oidc-portal` is the cloud exception: no
Makefile/k3d, validate-only in CI.

## What these are NOT

- Production-grade. They use the smallest cluster size, in-cluster databases, and
  demo credentials.
- A complete tutorial. They expect the reader to know Terraform + Helm basics.

## Module sources

Demos use **relative** sources (`../../terraform/<section>/<path>`) — see
[`AGENTS.md`](./AGENTS.md). They intentionally do *not* pin a `?ref=<tag>`: a demo
tracks the library in this checkout, so `terraform validate` runs offline and
always against the current modules. The `build-poc` skill pins a version when it
emits a standalone PoC for a prospect.
